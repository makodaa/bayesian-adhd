# Development Log - February 25, 2026

## Session Goal

Improve EEG visualization performance and memory usage, remove blocking behavior that waited for all bands, and move to a lazy, cache-backed visualization flow integrated into the main analysis workflow.

---

## Initial Problem Statement

The previous implementation generated all EEG visualizations in one heavy request and returned base64 payloads. This caused:

- High CPU and memory usage in backend rendering
- Large response payloads (base64 overhead)
- Delayed UI rendering (no first paint until all visualizations completed)

---

## Architecture Implemented

### 1) New Visualization Context + Lazy Image API

Added new endpoints:

- `POST /api/eeg_visualization_context`
  - Validates uploaded CSV
  - Creates/refreshes cached visualization context
  - Returns `context_id`, expiration, and available bands

- `GET /api/eeg_visualizations/<context_id>/<band>.png`
  - Returns PNG directly (not base64 JSON)
  - Uses cache when available (`X-Visualization-Cache: HIT|MISS`)
  - Generates and stores on first miss

Removed legacy endpoints:

- `/visualize_all_eeg`
- `/visualize_eeg/<eeg_type>`

### 2) Filesystem Visualization Cache

Added `VisualizationCacheService` in `backend/app/services/visualization_cache_service.py`:

- Content-addressed context creation keyed by clinician + file hash + profile
- CSV bytes persisted per context
- Band image caching per context/profile/band
- TTL expiration and max-size eviction
- Context access controls tied to `clinician_id`

Default cache policy configured in `backend/app/config.py`:

- `VIZ_CACHE_TTL_SECONDS = 21600` (6 hours)
- `VIZ_CACHE_MAX_BYTES = 1073741824` (1 GB)
- `VIZ_CACHE_DIR = /tmp/bayesian_adhd_viz_cache`

### 3) Lightweight EEG Preview Rendering

Refactored `backend/app/services/visualization_service.py`:

- Replaced heavy MNE plot path for UI previews with lightweight Matplotlib stacked traces
- Single filtering pass using SciPy SOS filters
- Downsampling to bounded points (`PREVIEW_MAX_POINTS`)
- Reduced DPI and controlled figure sizing
- PNG bytes returned directly

### 4) Frontend Lazy Loading by Tab

Updated `backend/app/templates/index.html`:

- No more bulk visualization request
- Visualization tabs stay disabled until context is ready
- `raw` band loads first after analysis
- Other bands load only when their tab is shown (`shown.bs.tab`)
- In-flight and loaded sets prevent duplicate requests

---

## Workflow Integration Fixes

During validation, a workflow gap was identified: visualization context creation was not guaranteed to be tied to the exact `/predict` upload in all cases.

Implemented fix in `backend/app/main.py`:

- `/predict` now reads upload bytes once
- Reuses same bytes for both dataframe processing and context creation
- Returns `result.visualization_context_id`

Frontend now prefers this context ID immediately, with fallback to explicit context creation only if missing.

---

## Debugging Notes

After rollout, user reported visualizations appeared inactive with no obvious network calls/logs.

Root causes and fixes:

1. **Network visibility issue**
   - Previous implementation used `new Image().src = ...`, which may not appear under Fetch/XHR tooling.
   - Switched band loading to explicit `fetch(...)` and blob/object URL rendering.

2. **Observability gap**
   - Added explicit backend logs for:
     - context creation start/success
     - predict-linked context assignment
     - visualization image request and cache HIT/MISS

3. **Resilience in frontend ratio display**
   - Guarded ratio formatting against non-numeric values to avoid UI interruption (`Number.isFinite` check before formatting).

After these fixes, user confirmed visualization flow works.
